11 Code quality and style
12 ----
13 
14 All code should be developed according to the [Google C++ style guide](https://google.github.io/styleguide/cppguide.html). In addition, the code should conform to the following guidelines:
15 
16  * Code should be covered by tests
17  - Wherever possible, automated testing should be used
18  - Tests cases should at least exercise all documented requirements
19  - If automated testing is not possible, manual test cases should be described
20  * It must be easy for a developer checking out a project to run the test suite based on the information in the [readme](README.adoc)
21  * All code must pass all unit tests before a merge request is made
22  - Tests that don't pass should be marked pending (with justification) or should be fixed.
23  * All code must pass formatting and static linting tests
24  * Features should be developed in feature branches
25  * Only working code should go into the master branch.
26  - master should always be in a deployable state
27  - Undeployable code should stay on feature branches
28  - Code failing the active unit or integration tests for a project is undeployable
29  - Functionally incomplete code is not necessarily undeployable
30  * Feature branches should only contain single features
31  - Developers should not create large, month-long, multiple-feature branches
32  - Developers should try to have their code merged at least once every week
33  * All code must be reviewed before it is merged to a master branch.
34  - The reviewer can not be the code author
35  - All the reviewer's concerns must be resolved to reviewer's satisfaction
36  - Reviews should enforce the coding guidelines for the project
37  * Bugs reported against code on a master branch should be reproduced with a failing test case before they are resolved
38  - Reviewers of code including bug fixes should check that a covering test has been included with the fix
39  * New features, bug fixes, and removed functionality should be documented in the [changelog](CHANGELOG.md)

41 Making a Pull Request
42 ----
43 
44 When you start developing a feature, please create a feature branch that includes the type of branch, the ID of the issue or ticket if available, and a brief description. For example `feat/9/https-support`, `fix/OTA-123/fix-token-expiry` or `refactor/tidy-up-imports`. Please do not mix feature development, bugfixes and refactoring into the same branch.
45 
46 When your feature is ready, push the branch and make a pull request. We will review the request and give you feedback. Once the code passes the review it can be merged into master and the branch can be deleted.
47 
48 Developer Certificate of Origin (DCO)
49 ----
50 
51 All commits in pull requests must contain a `Signed-off-by:` line to indicate that the developer has agreed to the terms of the [Developer Certificate of Origin](https://developercertificate.org) (see [readme](README.adoc) for more details). A simple way to achieve that is to use the `-s` flag of `git commit`.
52 
53 New pull requests will automatically be checked by the [probot/dco](https://probot.github.io/apps/dco/).
